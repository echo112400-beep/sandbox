Full Interaction Flow

🔄 完整交互流程

从创建 BatchSandbox 到 Pod 分配、Pool 自动扩缩的端到端流程

Step 1 — 用户创建 BatchSandbox

POST /v1/sandboxes 或 kubectl apply BatchSandbox

spec.replicas=3 · spec.template (含镜像+资源) · spec.poolRef 为空

Step 2 — BatchSandbox Controller: assignPool()

自动选池 → patch spec.poolRef

ProfileStore.GetProfile("default") → DefaultAssigner.AssignPool()
Predicate 链: Capacity ✓ → Image ✓ → Resource ✓ → NodeSelector ✓
Scorer ResBalance(LeastAllocated): 候选池中选负载最低的
→ patch spec.poolRef = "pool-python"

💡 如果 Sandbox 已有 spec.poolRef (手动指定)，跳过此步骤直接进入分配
Step 3 — Pool Controller 被触发 (Watch)

Pool Controller 监听到新 BatchSandbox → reconcilePool()

Watches BatchSandbox (filter: poolRef!="") · 检测到新 Sandbox 引用本 Pool

Step 4 — scheduleSandbox: 计算分配

Allocator.Schedule() → PackedSchedule 算法

SandboxRequest: allocated=0, replicas=3, supplement=3, toRelease=[]
availablePods: Pool 中 Ready 且未分配的 Pod = [p1, p2, p3, p4, p5]
→ ToAllocate: {sandbox: [p1, p2, p3]} · PodSupplement: 0 · ToRelease: {}

Step 5 — doAllocate: 持久化分配

Phase 1: 内存更新 → Phase 2: 注解持久化 (失败回滚)

① store.UpdateAllocation(p1→sbx, p2→sbx, p3→sbx)
② syncer.SetAllocation → patch BatchSandbox annotation:
  alloc-status: {"pods":["p1","p2","p3"]}
  finalizers: ["pool-allocation"]
③ 注解写入失败 → 回滚内存到旧状态

Step 6 — scalePool: 温度调节

buffer 超下限 → 扩容补足

状态: total=5 · allocated=3 · buffer=2 · bufferMin=3 → buffer < min!
desiredBuffer = (3+8)/2 = 5
desiredSchedulable = max(3+0+5, poolMin) = max(8, 5) = 8
createCnt = min(8-5, poolMax-5, maxUnavailable-notReady) = 2
→ createPoolPod() × 2

Core Mechanism

🌡️ 温度控制器 — Buffer 管理

四参数自动调节，滞回避免震荡，两重限速保护

📐
容量四参数模型
CapacitySpec
poolMin=5bufferMin=2allocated=8buffer=4bufferMax=8poolMax=50
allocated
total=12 ·  allocated=8 ·  buffer=4 ·  poolMax=50
  • bufferMin / bufferMax — 空闲缓冲窗口：低于 min 扩容，高于 max 缩容
  • poolMin / poolMax — 池绝对上下限：硬约束，不可突破
  • 回中策略 — 缓冲超范围时回到 (min+max)/2，非边界值
📈
scalePool 决策公式
每次 reconcile 执行
// Step 1: 评估当前缓冲
buffer = schedulable - allocated

// Step 2: 决定目标缓冲
if buffer < bufferMin or buffer > bufferMax:
  desiredBuffer = (bufferMin + bufferMax) / 2
else: desiredBuffer = buffer

// Step 3: 计算期望总数
desiredTotal = max(allocated + supply + desiredBuffer, poolMin)
maxNew = max(poolMax - totalPodCnt, 0)
limited = maxUnavailable - notReadyCnt
createCnt = min(need, maxNew, limited)
🛡️
两重扩容限速
  • 硬上限: PoolMax - totalPodCnt — 绝对不可超越
  • 软限速: maxUnavailable (默认 25%) — 如 Deployment 滚动更新节奏
  • notReady 计入: 未就绪 Pod 也计入不可用数 → 防过度扩容
  • ScaleExpectation: 追踪创建中的 Pod，避免 reconcile 竞态
  • 超时放弃: 如果 Expectation 未满足超过阈值 → 清除脏数据
📉
缩容策略
  • 优先级: 回收/升级标记 Pod > 空闲 Pod (最老优先)
  • toDeletePods: 回收 handler 决定删除的 Pod
  • idlePods: 未分配的空闲 Pod，按 CreationTimestamp 排序
  • 缩容到 desiredSchedulable 为止
Allocator

💾 分配器 — 双层状态 + 三阶段事务

内存快 + 注解持久 = 高性能 + Crash 可恢复

🧠
内存层
InMemoryAllocationStore
  • 数据结构: map[ns/poolName] → poolEntry
  • poolEntry: per-Pool sync.RWMutex + map[podName]sandboxName
  • 细粒度锁: 每 Pool 独立锁，不同 Pool 并发无竞争
  • sync.Map: 管理 pools 索引，高并发读
  • Double-Check Locking: getOrCreatePool() 中避免重复创建
📝
注解层
AnnoAllocationSyncer
  • alloc-status: {"pods":["p1","p2"]} — 当前分配
  • alloc-release: {"pods":["p1"]} — 正在回收中
  • alloc-released: {"pods":["p1"]} — 已回收完成
  • Finalizer: pool-allocation — 防 Sandbox 在有分配时被删
  • 注解存在 BatchSandbox 上，不依赖外部存储
🔄
Recover 恢复
sync.Once · 失败即退出
  • 时机: Controller 首次 Schedule() 触发
  • 流程: List 所有 BatchSandbox → 解析 alloc-status → 扣除 alloc-released → 重建内存
  • 一致性: 恢复失败直接 os.Exit(1) — 宁不可用也勿脏数据
  • 孤儿 GC: 恢复后 force-delete 的 Sandbox Pod 自动走回收路径
🔐
三阶段分配事务
保证内存与注解的一致性
操作Phase 1Phase 2失败处理
分配 Allocate 内存.更新 (乐观) 注解.持久化 (MergePatch + Finalizer) 注解失败 → 回滚内存 到旧状态
释放 Release 注解.持久化 (alloc-released) 内存.释放 (delete from map) 先持久化再释放 → Pod 不会被重复分配

Algorithm

🧮 PackedSchedule 贪心算法

默认调度算法 — 按 Sandbox 顺序依次满足，先到先得

⚡
算法伪代码
func PackedSchedule(availablePods, allRequest) AllocAction:
    action = {ToAllocate:{}, ToRelease:{}, PodSupplement:0}

    for each req in allRequest:
        // 1. 记录释放
        if len(req.ToRelease) > 0:
            action.ToRelease[name] = req.ToRelease

        // 2. 计算分配
        need = req.PodSupplement
        if need <= 0: continue

        if len(available) >= need:
            action.ToAllocate[name] = available[:need]
            available = available[need:]          // 切下分配
        elif len(available) > 0:
            action.ToAllocate[name] = available
            action.PodSupplement += need - len(available)
            available = nil                       // 用尽
        else:
            action.PodSupplement += need            // 完全缺额

    return action
📋
SandboxRequest 计算
// 正常 Sandbox
allocated = GetSandboxAllocation(sandbox)
released = GetSandboxReleased(sandbox)
releaseReq = alloc-release annotation
supplement = replicas - len(allocated)
toRelease = releaseReq - released (未完成的释放)

// 终止中的 Sandbox
supplement = 0 (不分配新 Pod)
toRelease = allocated - released (全部释放)

// 孤儿 Sandbox (force-deleted)
supplement = 0
toRelease = 已分配但 Sandbox 不在列表中的 Pod (GC)
Design Highlights

✨ 关键设计决策

🏗️
Annotation 驱动协作

BatchSandbox 和 Pool 两个 Controller 不直接通信 — 通过 CRD 注解实现松耦合协同。Watch Predicate 精确过滤必要事件。

🔄
sync.Once 恢复

Controller 重启时首次调度触发 Recover()，从注解重建内存态。失败直接 os.Exit(1) — 宁可不可用也不使用脏状态。

📉
滞回 (Hysteresis)

buffer 超出 [min,max] 时不纠正到边界，而是回到中点。避免边界附近的反复扩缩容震荡。

🔒
Finalizer 安全网

pool-allocation finalizer 阻止在有未释放 Pod 时删除 Sandbox。所有 Pod released 后才移除 finalizer。

🧹
孤儿 Pod GC

force-delete BatchSandbox 后，Allocator 自动检测缺失 Sandbox 的已分配 Pod，构造孤儿请求走正常回收路径。

🎛️
可插拔算法

Algorithm 接口支持 Packed / Spread 切换。Profile 机制支持自定义 Predicate + Scorer 组合。回收 Handler 支持 Delete/Restart/Noop。

Pause Flow

⏸️ Pause 完整流程

从用户发起暂停请求到沙箱完全冻结的端到端流程

Phase 0 — Server 端

POST /v1/sandboxes/{id}/pause

校验 phase 必须为 Succeed（Paused/Pausing/Resuming/Failed 均拒绝）→ patch spec.pause=true

💡 Retry Bridge: 如果上次 Pause 失败 (PauseFailed=true)，Server 先 patch nil 再 patch true，两次操作完成桥接
Phase 1 — dispatchPauseResume

spec.pause=true ∧ generation > pauseObservedGen → handlePause()

校验 replicas=1 → 停止 Task → ACK phase=Pausing, pauseObservedGen=gen

Phase 2 — ensureInternalPauseSnapshot

创建 SandboxSnapshot 子资源 (OwnerRef: BatchSandbox)

snapshotName = {sandboxName}-pause · spec.sandboxName = BatchSandbox.Name

Phase 3 — Snapshot Controller: handlePending

解析源 Pod → 计算 imageURI → buildCommitJob

imageURI = {registry}/{sandboxName}-{containerName}:snap-gen{generation}
通过 Pool 分配注解 / Label / 命名规则三种方式查找 Pod

Phase 4 — Commit Job (K8s Job)

image-committer 在源节点执行六步流程

① nerdctl ps 定位容器  →  ② nerdctl pause (FS 一致性)  →  ③ nerdctl commit (rootfs→OCI)  →  ④ nerdctl unpause (always!)  →  ⑤ nerdctl login + push  →  ⑥ nerdctl inspect (digest)

Phase 5 — Snapshot Controller: handleCommitting

读取 Job Pod 的 terminationMessage → 解析 digest → phase=Succeed

若 Job 失败 → best-effort unpause Job 恢复容器 → phase=Failed

Phase 6 — syncPauseOrClear → completePause

Snapshot Succeed → 删除 Pod / 脱离 Pool → phase=Paused

非 Pool 模式: 直接删除所有 Pod
Pool 模式: 从 Pod 固化模板 (去 Pool 标签) → 写回 spec.template → 清 poolRef → Pool GC 回收旧 Pod

Resume Flow

▶️ Resume 完整流程

从已暂停状态恢复到运行中

Phase 0 — Server 端

POST /v1/sandboxes/{id}/resume

校验 phase 必须为 Paused → patch spec.pause=false

Phase 1 — handleResume

ACK phase=Resuming + RequeueAfter 1s

不做业务逻辑，仅确认收到请求，交由下轮 reconcile 执行

Phase 2 — continueResume

从 Snapshot status 读取 imageURI → 替换 template 镜像

imageMap = {containerName: snapshotImageURI}
→ template.spec.containers[i].Image = imageMap[name]
→ 确保 imagePullSecret → Pool 模式清 poolRef

💡 若 Snapshot 不存在但已有 Ready Pod → 视为 resume 已完成（幂等保护）
Phase 3 — 正常 Reconcile 流程

scaleBatchSandbox() → 创建新 Pod (使用快照镜像)

persistRuntimeView() → phase=Succeed → ACK generation

Phase 4 — 清理

下次 reconcile: dispatchPauseResume → generation==observed → skip

Resume 成功后自动 deleteInternalPauseSnapshot()

Core Logic

⚙️ 5-Case 调度表

dispatchPauseResume() — 防止 Controller 自身 spec 变更被误判为新请求

优先级条件动作说明
🔴 最高 phase == Resuming continueResume() 恢复流程进行中 → 替换镜像 → 等待正常流更新 phase
🟠 高 phase == Pausing syncPauseOrClear() 暂停流程进行中 → 轮询 Snapshot 状态
🟡 中 gen > observed ∧ pause=true handlePause() 新暂停请求 → ACK + 创建 Snapshot
🟢 中 gen > observed ∧ pause=false handleResume() 新恢复请求 → ACK Resuming
⚪ 低 gen > observed ∧ pause=nil skip 无 pause 意图 → persistRuntimeView 中 ACK
Commit Job Internals

🔧 Image Committer 内部流程

用 nerdctl 直接操作 containerd，绕过 CRI 层

Step 1 — 定位容器

nerdctl ps 查找容器 ID

通过 K8s 标准标签筛选：io.kubernetes.pod.name + io.kubernetes.pod.namespace + io.kubernetes.container.name

Step 2 — 暂停容器

nerdctl pause {containerID}

冻结容器进程以保证文件系统一致性。暂停失败的容器会被追踪用于后续恢复。

Step 3 — 提交镜像

nerdctl commit {containerID} {registry}/{name}:{tag}

将容器 rootfs 保存为 OCI 镜像。标签：内部快照 snap-gen{generation}，公开快照 snap-{sha256}

Step 4 — 恢复容器 (ALWAYS!)

nerdctl unpause {containerID}

无论 commit 成功或失败，必须恢复容器。Signal handler + defer 双保险。

Step 5 — 推送镜像

nerdctl login + push --insecure-registry

从 /var/run/opensandbox/registry/config.json (Docker config) 读取凭证。自动判断是否使用 insecure registry。

Step 6 — 提取 Digest & 写入结果

nerdctl inspect --format '{{.Id}}'

JSON 写入 /dev/termination-log：{"containers":[{"name":"sandbox","image":"...","digest":"sha256:..."}]}

# Commit Job 的 SecurityContext
RunAsUser:                0                       # root, 需要访问 containerd.sock
AllowPrivilegeEscalation:  false                   # 禁止提权
Capabilities:             Drop: ["ALL"]          # 去除所有 capabilities
NodeName:                  sourceNodeName          # 调度到源 Pod 所在节点

Design Highlights

✨ 关键设计细节

让快照系统可靠、安全、可重试的核心机制

🔄
Retry Bridge
Server 端可重试机制
  • PauseFailed 后重试: 先 patch nil 清旧值 → 再 patch true
  • 若第二步抛异常但 read-back 已是目标值 → 视为成功
  • 否则重试一次 target patch
  • 两步桥接确保 Controller 看到 nil→true 的 generation 变化
🛡️
Generation 栅栏
防止误判
  • pauseObservedGeneration 记录已处理的 generation
  • Controller 自身修改 spec (固化模板、替换镜像) 会导致 gen++
  • 但不会误判为新 pause/resume — 因为进行中 phase 优先判断
  • completePause 中只在 pause=true 时 ACK，防止覆盖 resume 请求
🔐
万能恢复
容器 unpause 保证
  • 正常路径: Step 4 在 commit 后自动 unpause
  • 失败路径: commit 失败也执行 Step 4 unpause
  • 信号路径: SIGINT/SIGTERM → signal handler → resumeAll
  • Panic 路径: defer + recover → resumeAll
  • SnapshotController 在 Job 失败时额外创建 best-effort unpause Job
🏊
Pool 模式特殊处理
透明化暂停/恢复
  • Pause: sourcePodTemplateForPause() 从 Pod 提取模板 → 删 Pool 标签 → 写回 spec.template → 清 poolRef → Pool GC 回收旧 Pod
  • Resume: continueResume() 清 poolRef + 移除 finalizer → 作为普通 Pod 重建
  • Pause 后脱离 Pool，Resume 后独立运行
🔗
OwnerRef 级联
资源生命周期绑定
  • BatchSandbox → SandboxSnapshot → Commit Job 三级级联
  • 删除 BatchSandbox → 自动删 SandboxSnapshot → 自动删 Commit/Unpause Job
  • Finalizer 保证 Snapshot 删除前清理 Job
  • SandboxSnapshot 的 TTL (300s) 自动清理完成后的 Job Pod
📊
结果传递机制
Kubernetes Native
  • image-committer 将结果写入 /dev/termination-log
  • Controller 从 Pod Status containerStatuses[].state.terminated.message 读取
  • JSON 格式含 per-container name/image/digest
  • 无额外通信通道，纯 K8s 原语

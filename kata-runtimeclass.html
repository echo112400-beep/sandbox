Layer 1 — Configuration

⚙️ 配置层: SecureRuntimeConfig

TOML 中定义安全运行时，Pydantic 模型自动交叉校验

📋
模型定义
Python — config.py 
class SecureRuntimeConfig(BaseModel):
    type: Literal["", "gvisor", "kata", "firecracker"] = ""
    docker_runtime: Optional[str] = None      # e.g. "kata-runtime"
    k8s_runtime_class: Optional[str] = None    # e.g. "kata-qemu", "kata-fc"
✅
跨字段校验规则
typeDockerK8s约束
""必须 None必须 None完全禁用
"kata"至少一个至少一个Docker+K8s 双后端支持
"firecracker"忽略必填仅 K8s + AppConfig 级校验拦截 Docker

Layer 2 — Startup Validation

🚀 启动校验: Fail-Fast

服务启动时验证 RuntimeClass 存在，而非等请求时才报错

Python — runtime_resolver.py 
async def validate_secure_runtime_on_startup(config, k8s_client):
    if config.secure_runtime.type == "":
        return  # 未启用 → 跳过

    # K8s: 调用 API 验证 RuntimeClass 存在
    runtime_class = resolver.get_k8s_runtime_class()  # → "kata-vm-isolation"
    k8s_client.read_runtime_class(runtime_class)

    # 404 → ValueError("Configured Kubernetes RuntimeClass 'kata-vm-isolation' does not exist.")
    #     → 服务启动失败 (fail-fast)
    # 200 → log: "Kubernetes RuntimeClass 'kata-vm-isolation' is available."
Layer 3 — Runtime Resolution

🔀 运行时解析层: SecureRuntimeResolver

运维配置 → 后端运行时参数的映射器

🗺️
默认映射表
typeDocker --runtimeK8s RuntimeClass
"gvisor"runscgvisor
"kata"kata-runtimekata-qemu
"firecracker"N/Akata-fc
🔍
解析优先级
def get_k8s_runtime_class():
    if type == "":
        return None                   # 不使用安全运行时
    if k8s_runtime_class != None:
        return k8s_runtime_class        # 显式指定 → 直接使用
    return DEFAULTS[type]            # 未指定 → 查默认表

例: type="kata" 不指定 k8s_runtime_class → "kata-qemu"

💡 Provider 构造函数中一次性存储: self.runtime_class = self.resolver.get_k8s_runtime_class() — 后续所有 Pod 创建直接使用此值
Layer 4 — Pod Creation

🏗️ Pod 创建: runtimeClassName 注入

13 步构建 Pod Spec，runtimeClassName 在第 6 步注入

构建 initContainer — execd 二进制注入

从 execd_image 创建 init 容器，拷贝 execd 到共享卷

构建 mainContainer — 用户 image/entrypoint/env/resources

合并用户请求中的镜像、入口命令、环境变量、资源限制

组装 pod_spec = {initContainers, containers, volumes}

基础 Pod Spec 字典

应用 Windows profile (如适用)

覆盖整个 pod_spec 为 Windows 兼容格式

应用 platform nodeSelector

Linux → kubernetes.io/os: linux

★ 注入 runtimeClassName = "kata-qemu"

pod_spec["runtimeClassName"] = self.runtime_class ← 仅此一行完成注入

应用 imagePullSecrets

应用 egress sidecar (如有 network_policy)

应用 volumes

包装为 BatchSandbox CR spec

spec.template.spec = pod_spec

加载 template ConfigMap 并 merge_with_runtime_values()

回读 runtimeClassName → ensure_egress_runtime_compatible()

从合并后的 spec 中读取最终值做最后校验

k8s_client.create_custom_object(batchsandbox)

创建 BatchSandbox CR → Controller reconcile → 创建 Pod

Validation Chain

🔒 校验链: 9 个校验点

#位置校验内容
1config.pyTOML 加载: type/docker_runtime/k8s_runtime_class 交叉验证
2config.pyAppConfig: Firecracker + Docker → 拒绝
3main.py启动: 创建 K8s client, 调 validate_secure_runtime_on_startup()
4runtime_resolver.py_validate_k8s_runtime_class(): k8s_client.read_runtime_class(name) → 404 fail-fast
5runtime_resolver.py_warn_gvisor_egress_incompatibility(): 仅 warn gVisor, Kata 无警告
6batchsandbox_provider.py__init__: SecureRuntimeResolver.get_k8s_runtime_class() → 存储
7validators.pyensure_egress_runtime_compatible(): gVisor + networkPolicy → 403, Kata → 通过
8batchsandbox_provider.pycreate_workload(): pod_spec["runtimeClassName"] = self.runtime_class
9batchsandbox_provider.pyPost-merge: 回读 runtimeClassName → ensure_egress_runtime_compatible() 二次校验
⚠️ 关键: Kata + Egress = 兼容
仅 gVisor 在黑名单中 — 缺 iptables nat 导致 egress sidecar DNS 重定向无法工作。
Kata 是完整 VM，支持 iptables NAT，错误消息显式推荐用 Kata 替代 gVisor。

Design Decisions

🎯 关键设计决策

🔒
运维配置 · 用户不可选

一个 Server = 一个安全运行时 · 用户无法在 API 请求中切换 runc/kata/gVisor · 安全边界由运维定义

🔀
双后端支持 Kata

Docker → --runtime kata-runtime · K8s → runtimeClassName: kata-qemu · 同一配置切换

🚫
Firecracker 仅 K8s

config.py 模型级 + AppConfig 级双重拦截 · Docker 环境设 firecracker → ValueError

✅
Kata 兼容 Egress

仅 gVisor 缺 iptables NAT · error 消息推荐 kata 替代 · Kata 完整 VM 支持所有网络操作

🗺️
默认值可覆盖

type="kata" 默认 kata-qemu · 可指定 kata-vm-isolation (AKS) · 支持任意 RuntimeClass 名

⚡
Fail-Fast 启动校验

启动时读 RuntimeClass → 404 直接退出 · 而非等到第一个创建请求时才报错 · 配置错误早发现
